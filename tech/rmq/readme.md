### Слайд 1 – Как появились брокеры сообщений: от синхронных вызовов к асинхронности
- 1980‑е: удалённые вызовы процедур (RPC) – синхронная связь, жёсткая зависимость
- Рост распределённых систем → нужна асинхронная развязка
- Появление MOM (Message-Oriented Middleware): IBM MQ, TIBCO, MSMQ
- Конец 1990‑х: JMS (Java Message Service) – стандарт для Java, но без межъязыковой интероперабельности
- 2003–2006: разработка открытого протокола AMQP (JPMorgan, Cisco, Red Hat и др.)
- 2007: Rabbit Technologies выпускает RabbitMQ – первую популярную реализацию AMQP на Erlang
- Дальнейшая судьба: VMware → Pivotal → Broadcom, широкая экосистема (плагины, клиенты на всех языках)

### Слайд 2 – Какие проблемы решают брокеры сообщений
- **Временная развязка** – отправитель и получатель работают независимо во времени
- **Пространственная развязка** – отправителю не нужен адрес получателя, только брокер
- **Буферизация нагрузки** – очереди сглаживают пики трафика
- **Гарантии доставки** – персистентность, подтверждения, транзакционность
- **Маршрутизация и фильтрация** – брокер сам направляет сообщения нужным получателям
- **Отказоустойчивость** – если потребитель упал, сообщение остаётся в очереди и будет обработано позже
- **Масштабирование** – параллельные потребители на одной очереди (конкурирующие потребители)

### Слайд 3 – Архитектура RabbitMQ: ключевые компоненты
- **Broker** – сервер RabbitMQ, управляющий обменниками, очередями, соединениями
- **Virtual Host (vhost)** – изолированный контейнер для обменников, очередей, прав доступа (аналог базы данных в СУБД)
- **Connection** – долгоживущее TCP‑соединение между приложением и брокером
- **Channel** – «лёгкое» логическое соединение внутри одного TCP (экономия ресурсов, изоляция операций)
- Почему каналы: открывать множество TCP‑соединений дорого, каналы же дешёвы, работают мультиплексированно

### Слайд 4 – Модель маршрутизации AMQP 0-9-1
- Producer → **Exchange** → **Binding** → **Queue** → Consumer
- Producer всегда отправляет сообщение в Exchange, а не напрямую в очередь
- Exchange анализирует **Routing Key** сообщения и **Binding Key** очереди, чтобы решить, в какие очереди поместить сообщение
- Одно сообщение может попасть в несколько очередей (если binding позволяет)
- Потребитель забирает сообщение из очереди и подтверждает обработку (ack)

### Слайд 5 – Exchanges: типы и маршрутизация (часть 1 – Direct и Fanout)
- **Direct exchange**  
  - Маршрутизация по точному совпадению routing key и binding key  
  - Пример: ключ `"order.created"`, очередь `order_events` привязана с ключом `"order.created"` – сообщение попадёт в неё  
  - Если несколько очередей привязаны с одним ключом – все получат копию (как fanout)  
- **Fanout exchange**  
  - Игнорирует routing key, отправляет копию во **все** привязанные очереди  
  - Широковещательная рассылка (например, уведомление всем подписчикам об изменении конфигурации)

### Слайд 6 – Exchanges: типы и маршрутизация (часть 2 – Topic, Headers, Default)
- **Topic exchange**  
  - Routing key – строка из слов, разделённых точкой (`"store.order.created"`)  
  - Binding key может содержать `*` (ровно одно слово) и `#` (ноль и более слов)  
  - Пример: очередь привязана с ключом `"order.*"` – получает `order.created`, `order.paid`, но не `order`  
- **Headers exchange**  
  - Маршрутизация по заголовкам сообщения (`headers`), а не по routing key; условие `all` или `any`  
  - Используется реже из‑за накладных расходов  
- **Default exchange** (безымянный `""`)  
  - Неявный direct‑обменник, привязанный к каждой очереди автоматически; routing key = имя очереди  
  - Удобен для простейшей отправки напрямую в очередь

### Слайд 7 – Очереди: базовые свойства и декларация
- Очередь – именованный буфер, хранящий сообщения до момента доставки потребителю
- При объявлении задаются свойства:
  - **durable** – переживает перезапуск брокера (но сообщения также должны быть помечены persistent)
  - **exclusive** – доступна только одному соединению, удаляется при его закрытии
  - **auto‑delete** – удаляется, когда отключается последний потребитель
- Аргументы очереди (x-arguments):
  - **Message TTL** – время жизни сообщения в очереди (на уровне очереди или на уровне сообщения)
  - **Max length / max length bytes** – ограничение количества или суммарного объёма сообщений
  - **x-dead-letter-exchange** – обменник для «умерших» сообщений
  - **x-queue-mode** – lazy, default (classic)

### Слайд 8 – Типы очередей: Classic, Quorum, Lazy
- **Classic queues** (по умолчанию)  
  - Хранят сообщения в оперативной памяти, на диск сбрасывают при необходимости  
  - Поддерживают зеркалирование (mirroring) для высокой доступности (но зеркалирование устаревает)  
- **Quorum queues** (рекомендованы для высокой доступности)  
  - Реализованы на основе алгоритма Raft (как в etcd, Consul)  
  - Данные реплицируются на большинство узлов, устойчивы к сбоям  
  - Более строгие гарантии сохранности, но выше задержка  
- **Lazy queues**  
  - Сообщения сразу записываются на диск, в RAM хранятся только метаданные  
  - Экономия оперативной памяти при работе с очень большими очередями (миллионы сообщений)  
  - Включается аргументом `x-queue-mode: lazy` (или политикой)

### Слайд 9 – Dead Letter Queue (DLQ): причины и настройка
- Сообщение попадает в DLQ, если:
  - Потребитель отклонил его (`basic.reject` или `basic.nack` с `requeue=false`)
  - Истекло время жизни (TTL) сообщения
  - Очередь переполнена (max length) и сообщение вытеснено
- Настройка:
  - `x-dead-letter-exchange` – обменник для мёртвых писем
  - `x-dead-letter-routing-key` – ключ, с которым они будут переданы в этот обменник
- Можно организовать цепочку: основная очередь → DLQ‑обменник → DLQ‑очередь для анализа или повтора

### Слайд 10 – DLQ: практические сценарии
- **Отложенная обработка** (delayed retry)
  - Основная очередь имеет TTL, после истечения TTL сообщение уходит в DLX и попадает во «вторичную» очередь без TTL
  - Потребитель вторичной очереди может повторить обработку с задержкой
  - Альтернатива – плагин `rabbitmq_delayed_message_exchange`, не требует DLQ
- **Карантин «ядовитых» сообщений**
  - Если сообщение вызывает ошибку многократно, его отправляют в DLQ для ручного анализа и исправления
- **Аудит и логирование**
  - Все «умершие» сообщения перенаправляются в специальный сервис‑коллектор для разбора инцидентов

### Слайд 11 – Надёжность: publisher confirms и consumer acknowledgements
- **Publisher Confirms** (подтверждение публикации)
  - Включается через `channel.confirmSelect()`
  - Брокер асинхронно подтверждает, что сообщение получено и сохранено (или сообщает об ошибке)
  - Гарантирует, что сообщение не потеряно на этапе передачи от продюсера к брокеру
- **Consumer Acknowledgements**
  - После получения сообщения потребитель должен явно подтвердить (`basic.ack`) или отклонить (`basic.nack`, `basic.reject`)
  - При разрыве соединения неподтверждённые сообщения возвращаются в очередь (автоматический requeue)
  - Подтверждение можно делать вручную после завершения бизнес‑логики (at‑least‑once семантика)

### Слайд 12 – Prefetch и управление нагрузкой потребителя
- **Prefetch count** (QoS) – сколько неподтверждённых сообщений разрешено потребителю одновременно
- Позволяет реализовать **fair dispatch**: не давать быстрому потребителю забирать всё, пока медленный обрабатывает одно
- Пример: `channel.basicQos(1)` – потребитель получит следующее сообщение только после подтверждения предыдущего
- Балансировка между пропускной способностью и равномерностью загрузки

### Слайд 13 – Проблема двойной записи (dual write) и зачем нужен Transactional Outbox
- Типичный микросервисный сценарий:
  1. Обновить запись в базе данных (например, статус заказа)
  2. Отправить событие в RabbitMQ (например, `"order.placed"`)
- Проблема: эти два действия не атомарны
  - Если после обновления БД упадём до отправки – потеряем событие
  - Если отправим событие, но БД не обновится – консистентность нарушена
- Классические распределённые транзакции (XA) слишком тяжелы и не всегда поддерживаются

### Слайд 14 – Паттерн Transactional Outbox: как решить проблему
- **Идея**: вместо прямой отправки сообщения, записать его в специальную таблицу `outbox` **в той же транзакции**, что и бизнес‑данные
- Состав таблицы Outbox: id, aggregate_id, event_type, payload, status, created_at
- Отдельный процесс (Message Relay) периодически читает записи из Outbox и отправляет их в брокер, помечая как отправленные
- Таким образом гарантируется: либо оба действия (БД + событие) зафиксированы, либо ни одного
- Доставка сообщения становится **at‑least‑once** (может быть дубликат при сбое релея), потребители должны быть идемпотентными

### Слайд 15 – Реализация Outbox с RabbitMQ: подходы и пример на Spring
- **Подход 1: Polling publisher**
  - Планировщик (например, `@Scheduled`) опрашивает таблицу Outbox, отправляет неотправленные записи через `RabbitTemplate`, удаляет/помечает отправленные
- **Подход 2: Change Data Capture (CDC)**
  - Инструменты (Debezium, Kafka Connect) отслеживают изменения в журнале БД и транслируют их в RabbitMQ
  - Снижает нагрузку на приложение, но усложняет инфраструктуру
- **Пример на Spring Boot:**
  - В сервисе: `@Transactional` метод, внутри сохранение сущности и вставка в `Outbox` через JPA
  - `OutboxScheduler`: `rabbitTemplate.convertAndSend(exchange, routingKey, payload)`
- **Idempotency Key:** потребитель должен уметь обрабатывать дубликаты (например, по `messageId` из заголовков)

### Слайд 16 – Spring AMQP: AmqpTemplate и RabbitTemplate
- **AmqpTemplate** – интерфейс, абстрагирующий обмен сообщениями (как `JmsTemplate` для JMS)
  - Методы: `convertAndSend`, `receiveAndConvert`, `send`, `receive`
- **RabbitTemplate** – реализация `AmqpTemplate` для RabbitMQ
  - Добавляет специфичные для RabbitMQ возможности:
    - `convertAndSend(exchange, routingKey, object, messagePostProcessor)` – постобработка сообщения
    - `invoke` для синхронного RPC
    - Настройка `confirmCallback` и `returnCallback` для publisher confirms
    - Управление каналами, таймаутами
- Обычно в коде используют `RabbitTemplate` напрямую, но для тестирования можно замокать интерфейс `AmqpTemplate`

### Слайд 17 – Работа с RabbitTemplate на примерах
- Отправка сообщения в формате JSON:
  ```java
  rabbitTemplate.convertAndSend("orderExchange", "order.created", orderDto);
  ```
- Отправка с MessagePostProcessor (добавление заголовков):
  ```java
  rabbitTemplate.convertAndSend(ex, key, payload, msg -> {
      msg.getMessageProperties().setMessageId(UUID.randomUUID().toString());
      return msg;
  });
  ```
- Получение одного сообщения (pull):
  ```java
  OrderDto dto = (OrderDto) rabbitTemplate.receiveAndConvert("order.queue");
  ```
- Настройка подтверждений:
  ```java
  rabbitTemplate.setConfirmCallback((correlationData, ack, cause) -> { ... });
  ```

### Слайд 18 – Панель администратора RabbitMQ: включение и обзор
- Management Plugin – веб‑интерфейс и HTTP API для управления и мониторинга
- Включение: `rabbitmq-plugins enable rabbitmq_management`
- Доступ: `http://host:15672` (логин/пароль по умолчанию guest/guest, только localhost)
- Основные разделы:
  - **Overview**: графики по сообщениям, соединениям, очередям, потреблению памяти
  - **Connections/Channels**: список клиентских подключений
  - **Exchanges**: все обменники, их типы, привязки
  - **Queues**: список очередей со статистикой (ready, unacked, total), возможность добавления/просмотра сообщений

### Слайд 19 – Просмотр сообщений в очереди через Management UI
- В разделе **Queues** клик по имени очереди → вкладка **Get messages**
- Параметры:
  - Количество сообщений (по умолчанию 1)
  - Режим: **Requester** (сообщение удаляется из очереди, как при обычном потреблении) или **Nack requeue** (возвращается обратно)
  - Кодировка (auto, UTF‑8)
- Просмотр содержимого: показываются заголовки и тело сообщения (можно отобразить в виде plain text, base64)
- **Move messages** – ручное перемещение сообщений из одной очереди в другую (например, из DLQ обратно в рабочую)
- **Purge** – очистка очереди

### Слайд 20 – Типовые сценарии применения RabbitMQ (часть 1)
- **Микросервисы: асинхронная интеграция**
  - Сервис заказов публикует `OrderPlaced`, сервис уведомлений подписан → отправка email/SMS
  - Связь через брокер, сервисы не знают адреса друг друга
- **Фоновые задачи (work queues)**
  - Генерация отчётов, конвертация видео, начисление бонусов
  - Круговая балансировка (round‑robin) между воркерами
- **Event‑driven архитектура**
  - Обновление кэшей, синхронизация поискового индекса при изменении данных

### Слайд 21 – Типовые сценарии применения RabbitMQ (часть 2)
- **RPC (Request‑Reply)**
  - Клиент отправляет запрос во временную очередь ответа, указывает `replyTo` и `correlationId`
  - Сервер обрабатывает и отвечает в эту очередь
  - `RabbitTemplate` поддерживает `convertSendAndReceive`
- **Отложенная обработка**
  - Через TTL + DLQ или плагин delayed‑exchange
  - Повторная попытка обработки с возрастающей задержкой (exponential backoff)
- **Приоритетные очереди**
  - Аргумент `x-max-priority`, сообщения с большим приоритетом обрабатываются раньше

### Слайд 22 – Пример Spring Boot приложения: Producer и Consumer
- **Producer (фрагмент)**
  ```java
  @RestController
  public class OrderController {
      @Autowired private RabbitTemplate rabbitTemplate;
      @PostMapping("/orders")
      public void createOrder(@RequestBody Order order) {
          // бизнес-логика...
          rabbitTemplate.convertAndSend("orderExchange", "order.created", order);
      }
  }
  ```
- **Consumer**
  ```java
  @RabbitListener(queues = "orderQueue")
  public void handleOrder(Order order) {
      log.info("Processing order {}", order.getId());
      // обработка
  }
  ```
- Настройка в `application.yml` (connection, exchange, queue, binding) через `@Configuration` класс с `TopicExchange`, `Queue`, `Binding`

### Слайд 23 – Введение в Apache Kafka
- Создана в LinkedIn (2010) для сбора и доставки потоковых данных, позже передана Apache Foundation
- Модель: **распределённый журнал (log)**, а не очередь сообщений
- Сообщения записываются в топики, которые делятся на партиции
- Каждое сообщение внутри партиции строго упорядочено и имеет уникальный offset
- Потребители (consumer groups) читают данные, сохраняя смещение; могут перечитывать историю
- Хранение сообщений настраивается по времени или размеру, а не удаляются после обработки
- Основное преимущество: огромная пропускная способность (миллионы сообщений/с)

### Слайд 24 – Сравнение RabbitMQ и Apache Kafka (таблица)
| Характеристика | RabbitMQ | Apache Kafka |
|----------------|----------|--------------|
| Модель данных | Очереди (FIFO), сообщение удаляется после ack | Распределённый лог; сообщения хранятся заданное время |
| Протокол | AMQP 0-9-1, MQTT, STOMP | Собственный бинарный протокол |
| Маршрутизация | Гибкая: exchange + routing key + binding | Простая: ключ определяет партицию |
| Порядок сообщений | В одной очереди при одном потребителе | В пределах партиции |
| Модель доставки | Push (брокер сам отдаёт) или Pull | Pull (потребитель запрашивает пачки) |
| Подтверждение | ack/nack, сообщение удаляется | offset фиксируется потребителем |
| Повторное чтение | Только если сообщение не подтверждено и requeue | Да, можно перечитать с любого offset |
| Производительность | ~10-50 тыс. сообщ/с | >1 млн сообщ/с |
| Хранение | До ack (persistent – на диске) | Настраиваемый срок (часы, дни) |
| Типичные сценарии | Задачи, умная маршрутизация, RPC | Потоковая обработка, событийный журнал, аналитика, event sourcing |

### Слайд 25 – Когда выбрать RabbitMQ, а когда Kafka?
- **RabbitMQ** подходит:
  - Приоритеты, сложная маршрутизация (topic, fanout)
  - Классические очереди задач с нагрузкой, требующей быстрой реакции
  - Требуется взаимодействие с существующими протоколами (MQTT, AMQP)
  - RPC и синхронные ответы
  - Меньшие объёмы данных, но нужна тонкая настройка поведения очередей
- **Kafka** подходит:
  - Огромные потоки событий (терабайты в день), логи, IoT
  - Нужна возможность перечитывать историю (аудит, аналитика, машинное обучение)
  - Строгий порядок обработки событий в контексте одной сущности (ключ партиции)
  - Event sourcing и CQRS
- На практике они часто сосуществуют: Kafka для событийного потока, RabbitMQ для оперативных задач

### Слайд 26 – Итоги и дополнительные материалы
- RabbitMQ – надёжный, гибкий брокер сообщений с богатой моделью AMQP
- Ключевые понятия: Exchange, Queue, Binding, DLQ, publisher confirms, consumer ack
- Для гарантированной доставки в связке с БД используем Transactional Outbox
