02· История

Брокеры сообщений выросли из синхронных RPC-вызовов

1980s
RPC. Удалённые вызовы — синхронные, жёсткая связность.
1990s
MOM. IBM MQ, TIBCO, MSMQ — first message-oriented middleware.
2001
JMS. Стандарт для Java, без межъязыковой совместимости.
2003
AMQP. JPMorgan, Cisco, Red Hat начинают открытый протокол.
2007
RabbitMQ. Rabbit Technologies, реализация на Erlang.
2010
VMware → Pivotal. Корпоративный owner и широкая экосистема.
2023+
Broadcom. Плагины, клиенты на всех языках.
Главный сдвиг
От синхронного вызова «жди ответа здесь и сейчас» — к асинхронному «оставь сообщение, забери когда сможешь». Это разблокировало масштабирование и отказоустойчивость распределённых систем.

03· Зачем нужны брокеры

Брокеры решают семь проблем распределённых систем

01
Временная развязка
Producer и consumer работают независимо во времени.
02
Пространственная развязка
Producer не знает адрес consumer'а — только брокер.
03
Буферизация нагрузки
Очередь сглаживает пики трафика.
04
Гарантии доставки
Персистентность, подтверждения, транзакции.
05
Маршрутизация и фильтрация
Брокер сам решает, кому отдать сообщение.
06
Отказоустойчивость
Падение consumer'а не приводит к потере данных.
07
Горизонтальное масштабирование
Конкурирующие потребители на одной очереди.
Брокер — это словарь, которым мы будем пользоваться всю лекцию.

04· Архитектура

RabbitMQ состоит из четырёх ключевых компонентов

Broker
Сервер RabbitMQ. Хранит обменники, очереди, соединения, политики.
Virtual Host (vhost)
Изолированный namespace. Аналог базы данных в СУБД — свои exchanges, queues, ACL.
Connection
Долгоживущее TCP. Один на приложение. Дорого открывать.
Channel
Лёгкое логическое соединение внутри TCP. По одному на поток. Дёшево, мультиплексируется.

06· Типы exchange · 1/2

Direct и Fanout: точное совпадение и широковещание

direct Маршрутизация по точному совпадению
Routing key сообщения == binding key очереди. Иначе сообщение игнорируется этой очередью.
Если две очереди привязаны с одним ключом — обе получат копию.
fanout Игнорирует routing key
Копия — во все привязанные очереди. Чистое широковещание.
Уведомить всех: сброс кэша, реиндексация, broadcast конфигурации.

07· Типы exchange · 2/2

Topic, Headers и Default — гибкая маршрутизация и shortcut

topic
Routing key из слов через точку. Шаблоны: * — ровно одно слово, # — ноль или более.
key = "store.order.created"
"store.#"  ✓
"*.order.*"  ✓
"order.*"  ✗
"#.created"  ✓
Самый используемый тип на практике.
headers
Маршрутизация по headers, не по routing key. Условие x-match: all или any.
headers:
  format = "pdf"
  type = "report"
binding (all):
format=pdf, type=report
Накладные расходы выше — используется редко.
default ""
Безымянный direct. К нему автоматически привязана каждая очередь с binding key, равным её имени.
// shortcut: прямо в очередь
channel.basicPublish(
  "",
  "order.queue",
  props, body);
Удобно для прототипов и точечной отправки.

08· Очереди · свойства

Очередь — это буфер с настраиваемыми свойствами

Флаги декларации
durable
сохраняется при рестарте брокера
Сама очередь — да. Сообщения — только если помечены persistent.
exclusive
только одно соединение
Удаляется при закрытии. Типично — для приватной reply-очереди в RPC.
auto-delete
удаляется без consumer'ов
Когда отключился последний потребитель — очередь исчезает.
x-arguments
// declare with arguments Map<String,Object> args = new HashMap<>(); args.put("x-message-ttl", 60000); // 60 sec args.put("x-max-length", 10_000); // items args.put("x-max-length-bytes", 10_485_760); // 10 MB args.put("x-dead-letter-exchange", "dlx.orders"); args.put("x-dead-letter-routing-key", "dead"); args.put("x-queue-mode", "lazy"); // or classic channel.queueDeclare("order.queue", true, false, false, args);
Все аргументы могут также задаваться policy на уровне vhost — переопределяя индивидуальные декларации.

09· Типы очередей

Classic, Quorum и Lazy — три способа хранить сообщения

по умолчанию
Classic
Сообщения в RAM, сбрасываются на диск по необходимости.
  • Mirroring (зеркалирование) — deprecated
  • Самая низкая латентность
  • Подходит для большинства задач
HA · рекомендовано
Quorum
Алгоритм Raft. Реплицируется на большинство нод кластера.
  • Гарантии сохранности при потере узла
  • Выше задержка — кворум по нодам
  • etcd · Consul · Kubernetes — тот же подход
экономия RAM
Lazy
Сообщения сразу на диск. В памяти — только метаданные.
  • Миллионы сообщений без OOM
  • Чуть выше латентность доступа
  • x-queue-mode: "lazy"

10· Dead Letter Queue

Dead Letter Queue ловит «умершие» сообщения

Настройка
// в аргументах основной очереди args.put("x-dead-letter-exchange", "dlx.orders"); args.put("x-dead-letter-routing-key", "order.dead");
DLX — обычный exchange. Куда дальше пойдёт сообщение — определяется bindings, как у любого другого обменника.
Можно строить цепочку: основная → DLX → DLQ → ещё одна DLX для retry-логики.

11· DLQ на практике

DLQ на практике: ретраи, карантин и аудит

сценарий 01
Delayed retry
Основная очередь с TTL. По истечении сообщение уходит в DLX и попадает во «вторичную» очередь — где его обработают с задержкой.
retry.queue (TTL=30s)
  ↓ expire
DLX → work.queue
Альтернатива — плагин rabbitmq_delayed_message_exchange.
сценарий 02
Карантин «ядовитых» сообщений
Сообщение валит обработчик исключением. После N неуспехов — nack(requeue=false), и в DLQ для ручного разбора.
attempts > 3
  → nack false
  → poison.queue
Не блокирует обработку остальных сообщений.
сценарий 03
Аудит и логирование
Все мёртвые сообщения уходят в сервис-коллектор. Складываются в ELK или БД для разбора инцидентов.
DLX → audit.queue
  → ELK / Postgres
  → alerts
Никогда не теряйте мёртвые сообщения молча.

12· Надёжность

Надёжность строится на подтверждениях с двух сторон

PUBLISHER CONFIRMS
channel.confirmSelect(). Брокер асинхронно подтверждает приём. Сообщение не теряется на пути producer → broker.
CONSUMER ACKNOWLEDGEMENTS
basic.ack только после бизнес-логики. Разрыв соединения — автоматический requeue. Семантика: at-least-once.

14· Проблема двойной записи

Двойная запись ломает консистентность БД и брокера

А распределённая транзакция?
XA (two-phase commit) теоретически решает проблему.
  • Тяжёлый протокол, требует координатора транзакций
  • Не поддерживается многими брокерами «из коробки»
  • Снижает доступность и пропускную способность
  • В микросервисной архитектуре — практически не используется
Нужен подход проще, дешевле и без распределённого консенсуса.

24· Apache Kafka

Apache Kafka — это распределённый журнал, а не очередь

  • LinkedIn, 2010 → Apache Foundation
  • Модель — append-only log, не очередь
  • Topic делится на партиции, в каждой строгий порядок и offset

  • Пропускная способность — миллионы msg/s

25· Сравнение

RabbitMQ vs Kafka — сравнение по девяти параметрам

Характеристика RabbitMQ Apache Kafka
Модель данных Очереди (FIFO), сообщение удаляется после ack Распределённый лог, сообщения хранятся retention period
Протокол AMQP 0-9-1 · MQTT · STOMP Собственный бинарный протокол
Маршрутизация Гибкая: exchange + routing key + binding Простая: ключ определяет партицию (hash)
Порядок сообщений В одной очереди при одном потребителе В пределах партиции, всегда
Модель доставки Push (или Pull) Pull (батчи)
Подтверждение ack / nack — сообщение удаляется Фиксация offset потребителем
Повторное чтение Только при requeue Да — с любого offset
Производительность ~10–50 тыс. msg/s >1 млн msg/s
Типичные сценарии Task queues · умная маршрутизация · RPC Streaming · event sourcing · аналитика · audit log

26· Выбор инструмента

Выбор между RabbitMQ и Kafka зависит от задачи

R
когда выбрать
RabbitMQ
  • Сложная маршрутизация (topic, fanout)
  • Очереди задач с быстрой реакцией
  • Приоритеты, retry, дедлайны
  • Совместимость с MQTT / legacy AMQP
  • RPC и синхронные ответы
  • Меньшие объёмы, тонкая настройка очередей
K
когда выбрать
Kafka
  • Огромные потоки событий — терабайты в день
  • Логи, IoT-телеметрия, метрики
  • Повторное чтение истории (аудит, ML, аналитика)
  • Event sourcing, CQRS, материализованные view'ы
  • Строгий порядок в рамках одной сущности (ключ партиции)
  • Backbone для интеграции десятков сервисов
На практике они часто сосуществуют. Kafka — backbone потока событий и аналитики. RabbitMQ — оперативные задачи, RPC, тонкая маршрутизация.
